# xzawedPAIS 플랫폼 비전 — 유기적으로 협업하는 AI 개발 조직

- 작성일: 2026-06-01
- 상태: 비전 확정용 (북극성 문서)
- 성격: 이 문서는 플랫폼의 **방향을 못 박는 상위 비전**이다. 개별 구현은 이 비전에서 갈라져 나오는 서브프로젝트로 진행하며, 각 서브프로젝트는 별도 설계 문서를 갖는다.

## 0. 한 줄 비전

> 능동적인 AI 에이전트들이 — Manager의 조율 아래 서로 **교차 검증**하며 — 공유 위키에 **도메인 지식을 누적**해 가는, **실제 개발회사처럼 유기적으로 협업하는 조직**. 사용자는 그 위의 PO(의사결정자)로서 핵심 분기를 승인한다.

## 1. 전체 그림

```
                      사용자 (PO)
                  ⛔ 핵심 분기마다 승인 / 수정요청 / 중단
                     (기본 수동, 게이트별 자동 전환 가능)
                          │
                   [Orchestrator]  사용자 의도 정제
                          │
                   [Manager = 팀리드]  능동적 조율·오케스트레이션
            ┌────────────┼────────────┐
        [기획]━━━━━[디자인]━━━━━[개발]━━━━━[테스트]━━[빌드]━━[보안]
            └──── 교차 확인 (왜곡 방지·전달 검증) ────┘
                          │
                  [공유 위키]  도메인 지식 누적·참조 → 작업할수록 강화
                          │
        빌드/실행: VSCode(오픈소스) 내장 환경  ·  GitHub 배포(승인 게이트)
```

## 2. 핵심 원칙

1. **능동성** — 에이전트는 수동적 도구가 아니라, 자기 전문 영역을 능동적으로 수행하고 필요하면 다른 전문가의 입력을 먼저 요청한다.
2. **유기적 협업** — 단방향 파이프라인이 아니라, 에이전트들이 서로 묻고 답하며 작업을 다듬는다(실제 개발팀처럼).
3. **교차 검증으로 왜곡 방지** — Manager가 A의 산출물을 B에게 전달할 때, B는 받은 내용이 A의 의도와 맞는지 **직접 확인**할 수 있어야 한다. 전달의 정확성을 검증·검토한다.
4. **도메인 지식 누적** — 산출물·결정·소통이 공유 위키에 기록되어, 프로젝트가 진행될수록 도메인 지식이 강화되고 이후 작업의 품질이 올라간다.
5. **사용자는 PO** — 사용자는 매 작업을 일일이 지시하지 않는다. 핵심 분기에서 승인/수정/중단으로 방향을 정하고, 나머지는 에이전트 조직이 자율 수행한다.
6. **결정론적 통제** — "기본 수동 승인" 같은 안전장치는 LLM의 자율 판단이 아니라 코드로 보장한다(건너뛸 수 없게).

## 3. 구성 요소 (현재 상태 → 목표)

| # | 요소 | 현재 상태 | 목표 |
|---|---|---|---|
| 1 | **능동적 에이전트** | 에이전트는 Manager가 부르면 일하고 결과만 반환하는 수동 도구 | 에이전트가 능동 수행 + 다른 전문가 입력을 먼저 요청 |
| 2 | **Manager 조율** | Claude tool-calling 루프가 다음 도구를 동적 선택(기반 존재) | 조율을 유지하되 협업·검증·위키를 엮는 팀리드로 강화 |
| 3 | **에이전트 교차 확인** | 없음 — 별(star) 토폴로지, 모든 통신이 Manager 경유, 에이전트 간 직접 확인 불가 | Manager 중계의 왜곡을 받는 에이전트가 검증할 수 있는 확인 경로 |
| 4 | **공유 위키(도메인 지식)** | 없음 | 산출물·결정·소통을 누적 기록·참조하는 도메인 지식 저장소 |
| 5 | **사용자 승인 게이트** | 없음 — 자동 진행. `info_request` 인프라만 존재 | 각 단계 전이마다 결정론적 게이트(기본 수동 + 게이트별 자동), 승인/수정/중단 |
| 6 | **데모 시연** | ✅ 구현(P4) — design_ui 승인 카드에 UISpec 인터프리터로 정적 목업 렌더 | 구현 전 정적 목업을 사용자에게 보여주고 승인 |
| 7 | **GitHub 배포 + 승인** | deploy-project 핸들러는 있으나 승인 게이트 없음 | 배포 전 사용자 승인 게이트 |
| 8 | **VSCode 내장** | 없음 | 오픈소스 VSCode를 내장해 빌드/실행 IDE 환경 제공 |

## 4. 현재 아키텍처의 핵심 갭 (코드 근거)

- **별 토폴로지**: 통신 규칙이 `manager:to-{agent}:{sessionId}` / `{agent}:to-manager:{sessionId}`뿐. 에이전트 간 직접 스트림이 없어 교차 확인·협업 불가.
- **수동 에이전트**: `redis-agent-handler.ts`가 에이전트 결과를 받아 즉시 반환. 에이전트가 "다른 전문가 입력 필요"를 능동 요청하는 경로 없음.
- **자동 진행**: Manager 시스템 프롬프트가 `"Use the available tools to fulfill the task request"`뿐(`runner.ts:243`)이라, 계획 후 사용자 확인 없이 Claude가 다음 도구를 자율 호출.
- **지식 휘발**: 세션이 끝나면 산출물·결정이 누적되지 않음. 도메인 지식 저장소 부재.
- ~~**데모 미표시**: `chat.store.ts`가 `uiSpec`을 저장만 함.~~ → ✅ **P4 구현 완료**: design_ui 승인 카드가 `UiSpecPreview`(Spec 인터프리터)로 정적 목업을 렌더.

재사용 가능한 자산: `info_request`/`waitForInfo`/`resolveInfo`/`clarificationContext`(사용자 대기·재개·재실행 인프라)는 이미 완비.

## 5. 로드맵 (분해와 의존 순서)

비전이 거대하므로 서브프로젝트로 분해한다. 각 서브프로젝트는 독립 설계·구현 사이클을 갖는다.

| Phase | 서브프로젝트 | 내용 | 의존 |
|---|---|---|---|
| **P1** | **협업 코어** | Manager 조율 강화 + 에이전트 교차 확인(왜곡 방지) + 능동적 입력 요청 | — (기반) |
| **P2** | **공유 위키** | 도메인 지식 누적·참조 저장소. 에이전트 산출물·결정 기록, 이후 작업에서 참조 | P1과 병행 가능 |
| **P3** | **승인 게이트** | 각 단계 결정론적 게이트(기본 수동 + 게이트별 자동), 승인/수정/중단 | P1 |
| **P4** ✅ | **데모 시연** | Designer UISpec → 정적 목업 렌더링 + 게이트 연동 (Spec 인터프리터, [설계](2026-06-05-p4-demo-preview-design.md)) | P3 |
| **P5** | **배포 게이트** | GitHub 배포 전 승인 | P3 |
| **P6** | **VSCode 내장** | 오픈소스 VSCode 빌드/실행 환경 | 독립 |

**첫 사이클**: 사용자가 "협업 모델부터"를 택했으므로 **P1(협업 코어)** 부터 상세 설계한다. P2(위키)는 P1의 교차 확인·지식 흐름과 밀접하므로 함께 검토한다.

이미 작성된 [승인 게이트 설계(A1)](2026-06-01-manager-approval-gates-design.md)는 본 로드맵의 **P3**에 해당하며, P1 이후로 재배치한다.

## 6. 성공의 모습 (북극성 시나리오)

"쇼핑몰 장바구니 기능 만들어줘" 한 줄 입력 시:
1. **기획**이 요구를 분석해 단계를 세우고, PG 선택 등 핵심 결정을 사용자에게 확인 → ⛔ 승인
2. **디자인**이 기획을 받아 UI를 설계하다 "실시간 재고 표시가 가능한가?"를 **개발에게 능동 질문** → 답을 반영 → 정적 목업을 사용자에게 시연 → ⛔ 승인
3. **개발**이 기획·디자인을 구현하며 받은 요구가 맞는지 **기획에게 교차 확인**, 테스트가 필요한 부분을 **테스트에게 요청**
4. **테스트**가 버그를 찾아 **개발에 피드백** → 수정 → 재검증
5. 모든 산출물·결정·교차 확인이 **공유 위키에 기록**되어, 다음 작업에서 이 도메인 지식이 참조됨
6. 빌드/실행은 **내장 VSCode 환경**에서, 완료된 결과는 사용자 승인 후 **GitHub에 배포**

이 흐름이 코드로 보장되고(승인은 건너뛸 수 없고), 에이전트가 능동·협업·검증하며, 지식이 누적되면 비전이 실현된 것이다.

## 7. 비범위 (이 문서에서 정하지 않는 것)

- 각 서브프로젝트의 상세 구현(메시지 스키마, 클래스, 테스트)은 해당 서브프로젝트 설계 문서에서 정한다.
- 위키의 저장 기술(파일/DB/벡터검색 등), 교차 확인의 정확한 프로토콜, VSCode 내장 방식(code-server vs OpenVSCode Server 등)은 각 Phase 설계에서 결정한다.
