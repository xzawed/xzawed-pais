# xzawedOrchestrator — Product Requirements Document

**버전:** 0.1
**날짜:** 2026-05-15
**범위:** xzawedOrchestrator (지휘자) 단독 구현

---

## 1. 목적

사용자가 자연어로 서비스 구현·관리 의도를 전달할 수 있는 Electron 기반 데스크탑 앱과, 해당 의도를 정제하여 xzawedManager에 전달하는 백엔드 서비스를 구현한다.

---

## 2. 사용자 및 시나리오

### 대상 사용자

- 소프트웨어 서비스를 만들고 싶지만 전체 개발 과정을 직접 수행하기 어려운 개인·팀
- xzawed 에이전트 시스템을 통해 서비스를 생성·운영하는 사용자
- 단일 사용자(개인) 또는 팀 단위로 사용

### 핵심 시나리오

**S1. 새 서비스 생성 요청**
> 사용자가 "쇼핑몰 서비스 만들어줘. 상품관리와 결제가 필요해"라고 입력하면, 지휘자가 요구사항을 구조화하여 총관리자에게 전달하고 진행 상황을 채팅으로 알려준다.

**S2. 추가 정보 수집**
> 지휘자가 사용자 의도를 파악하는 과정에서 구체적 정보가 필요하면, 우측 패널에 양식(콤보박스·체크박스 등)을 생성하여 구조화된 입력을 받는다.

**S3. 병렬 서비스 진행**
> 사용자가 Electron 창을 여러 개 열어 쇼핑몰 개발과 랜딩 페이지 디자인을 동시에 요청하고, 각각 독립적으로 진행된다.

**S4. 서비스 중단 후 재개**
> 에이전트 서비스 중 하나가 핫픽스 배포로 잠시 중단되어도, 진행 중이던 작업은 Redis Streams에 보관되어 재시작 후 자동으로 이어진다.

**S5. 팀 협업**
> 팀원들이 공유 서버에 각자의 Electron 앱으로 접속하여 별도 세션에서 독립적으로 작업을 진행한다.

---

## 3. 기능 요구사항

### 3.1 Electron 데스크탑 앱

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| F-APP-01 | 지휘자와 메신저 형태의 채팅 채널을 제공한다 | 필수 |
| F-APP-02 | 서버가 요청하면 우측 패널에 동적 UI를 렌더링한다 (양식, 목업, 현황판) | 필수 |
| F-APP-03 | 좌측 사이드바에 채널 목록과 진행 중 작업 목록을 표시한다 | 필수 |
| F-APP-04 | Claude 응답을 실시간 스트리밍으로 표시한다 | 필수 |
| F-APP-05 | Settings 화면에서 서버 URL·Claude 모드를 변경할 수 있다 | 필수 |
| F-APP-06 | `MODE=local` 시 Electron이 서버를 child process로 자동 기동한다 | 필수 |
| F-APP-07 | 여러 창을 동시에 열어 독립 세션을 운영할 수 있다 | 필수 |

### 3.2 백엔드 서버

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| F-SRV-01 | REST API로 세션 생성·메시지 전송·이력 조회를 제공한다 | 필수 |
| F-SRV-02 | WebSocket으로 Claude 응답 및 태스크 상태를 실시간 푸시한다 | 필수 |
| F-SRV-03 | MCP 서버 프로토콜로 세션 관리 도구를 외부에 노출한다 | 필수 |
| F-SRV-04 | Claude CLI(구독)·API 키·외부 서버 CLI 세 가지 실행 모드를 지원한다 | 필수 |
| F-SRV-05 | Redis Streams로 xzawedManager와 비동기 메시지를 주고받는다 | 필수 |
| F-SRV-06 | 세션별 독립 스트림으로 사용자 간 작업을 격리한다 | 필수 |
| F-SRV-07 | ACK 기반 처리로 서비스 중단 시 메시지를 보관하고 재시작 후 재개한다 | 필수 |
| F-SRV-08 | `MODE=local/remote`, `AUTH=none/jwt` 설정으로 배포 방식을 전환한다 | 필수 |

### 3.3 Claude 오케스트레이터 동작

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| F-ORC-01 | 사용자의 자연어 입력에서 서비스 구현 의도를 파악한다 | 필수 |
| F-ORC-02 | 필요 시 동적 UI 양식 스펙을 생성하여 추가 정보를 수집한다 | 필수 |
| F-ORC-03 | 수집된 요구사항을 구조화하여 xzawedManager용 메시지로 정제한다 | 필수 |
| F-ORC-04 | Manager 회신을 해석하여 사용자에게 자연어로 전달하거나 추가 질의한다 | 필수 |
| F-ORC-05 | CLI 모드 미설치 시 자동으로 API 모드로 폴백한다 | 선택 |

## 4. 비기능 요구사항

### 4.1 가용성 및 안정성

| ID | 요구사항 |
|----|----------|
| NF-01 | 에이전트 서비스 중단(핫픽스 배포) 시 진행 중 작업이 유실되지 않아야 한다 |
| NF-02 | 서비스 재시작 후 Redis Streams의 미처리 메시지를 자동으로 재개해야 한다 |
| NF-03 | 단일 사용자가 동시에 N개 창을 열어 N개 작업을 병렬로 진행할 수 있어야 한다 |

### 4.2 성능

| ID | 요구사항 |
|----|----------|
| NF-04 | Claude 응답은 첫 토큰이 2초 이내에 스트리밍되어야 한다 |
| NF-05 | REST API 응답(세션 생성, 메시지 수신)은 200ms 이내여야 한다 |

### 4.3 보안

| ID | 요구사항 |
|----|----------|
| NF-06 | `MODE=remote`일 때 HTTPS + WSS 연결을 강제해야 한다 |
| NF-07 | `AUTH=jwt` 모드에서 인증되지 않은 API 요청은 401을 반환해야 한다 |
| NF-08 | API 키·SSH 키 등 민감 정보는 환경변수로만 관리하며 코드에 포함하지 않는다 |

### 4.4 운영

| ID | 요구사항 |
|----|----------|
| NF-09 | 동일 코드베이스로 로컬·개인서버·팀서버 세 배포 환경을 지원해야 한다 |
| NF-10 | `GET /health`로 서버 상태를 확인할 수 있어야 한다 |

## 5. 인터페이스 요구사항

### xzawedManager 연동 (Redis Streams)

지휘자 → 매니저 스트림: `orchestrator:to-manager:{sessionId}`
매니저 → 지휘자 스트림: `manager:to-orchestrator:{sessionId}`

상세 메시지 포맷: [Redis Streams 메시징](../concepts/redis-streams.md) 참고
